POC · Talent Pool Scraping

Captura diaria de candidatos con indicador activo de búsqueda.

El flujo procesa listados públicos de LinkedIn e Indeed restringidos a una lista configurable de países fuera de Estados Unidos. Persiste únicamente perfiles con indicador "Buscando empleo" confirmado por el portal de origen y los deja consultables para el equipo.

Regla principal

Indicador de búsqueda activa requerido para inserción.

Antes de cualquier escritura en el pool, el flujo valida que el perfil declare explícitamente estar buscando empleo en su portal de origen. Sin esa señal, el registro se descarta. El campo open_to_work en la tabla talent_pool.candidates es NOT NULL y solo persiste con valor true.

Filtro bloqueante. El flujo descarta cualquier perfil sin la señal del portal de origen antes de la operación de escritura. Se loguea la tasa de descarte por proveedor y portal en execution_logs.
Señal por portal
🟢
Buscando empleo
Indicador formal en cada portal
LinkedIn Badge oficial "Open to Work" en el perfil del candidato
Indeed Estado "Actively looking for work" declarado por el candidato

Si un perfil no muestra este indicador explícitamente, queda fuera del pool. Sin excepciones.

Mecanismos de scraping

Dos proveedores disponibles para la extracción de listados.

Ambos siguen el mismo patrón de invocación HTTP usado hoy en el scraping de vacantes (general-us-openings). Pueden correr en paralelo durante la evaluación; tras la validación, se asigna el proveedor por portal en la configuración del flujo.

Bright Data API
Opción A

Plataforma de scraping con datasets pre-construidos por portal. Acceso vía Web Scraper API HTTP estándar — no MCP.

  • Endpoint: Web Scraper API + datasets de LinkedIn / Indeed
  • Modo: trabajos asíncronos, respuesta JSON estructurada
  • Filtros nativos: país, rol, indicador Open to Work en LinkedIn
  • Parsing: delegado al proveedor (sin XPath custom)
Oxylabs API
Opción B

Mismo patrón usado hoy en general-us-openings. Request con source: universal y parsing_instructions definidas por el equipo.

  • Endpoint: /v1/queries con source: universal
  • Modo: request síncrono o asíncrono con parsing_instructions
  • Filtros: construcción de URL del portal (LinkedIn search, Indeed Resume) por país
  • Parsing: XPath/CSS selectors mantenidos por el equipo
Flujo end-to-end

Plan diario parametrizado, extracción paralela y filtro ancla bloqueante.

El flujo lee su configuración de una tabla de parámetros antes de cada corrida. Procesa los portales con cualquiera de los dos proveedores disponibles y solo persiste candidatos con indicador "Buscando empleo" confirmado por el portal de origen.

Q1
Plan diario por país
Lee la lista configurada y selecciona los países que corresponden al día según la rotación semanal.
Q2
Construcción de búsquedas
Genera las URLs por (portal × país × filtros) que serán enviadas al proveedor de scraping asignado.
Q3 · Extracción (dos opciones de proveedor)
Branch A · Bright Data API
Q3a
Web Scraper API + datasets
Envía las URLs al proveedor y recibe los listados en JSON estructurado. Parsing delegado.
Branch B · Oxylabs API
Q3b
source: universal + parsing
Mismo patrón que general-us-openings. URLs enviadas con parsing_instructions custom.
Q4
Filtro ancla · Buscando empleo
Bloqueante. Solo pasan perfiles con la señal confirmada por el portal de origen. Tasa de descarte logueada por proveedor.
Q5
Dedupe
Unicidad por (source, source_id). Fallback por profile_url.
Q6
Normalize
Parseo de ubicación, inferencia de seniority, normalización de habilidades.
Q7
Persist + log
UPSERT en talent_pool.candidates. execution_id propagado a execution_logs.
